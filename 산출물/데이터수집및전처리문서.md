# 데이터 수집 및 전처리 문서

## 1. 개요

### 1.1 프로젝트 소개

치매알람은 치매 초기 증상 안내 및 지역 치매안심센터 정보 제공을 위한 LangGraph 기반 RAG 챗봇이다. 보호자와 사용자가 자연어로 질문하면, 구조화된 지역/센터 정보(GraphDB)와 비정형 증상·가이드라인 문서(VectorDB) 두 종류의 데이터를 검색해 답변을 생성한다.

### 1.2 데이터 구성 개요

| 구분 | 데이터 | 형태 | 용도 |
|---|---|---|---|
| GraphDB | 전국 치매안심센터 정보 (2개 공공데이터셋 병합) | 구조화 (CSV → Neo4j) | 지역/운영기관/프로그램 기준 센터 조회 |
| VectorDB | 치매 조기검진·관련 법령 가이드라인 | 비정형 문서 (웹페이지 → 텍스트) | 증상/검진 절차 관련 의미 기반 검색 |

## 2. GraphDB 데이터 — 치매안심센터 정보

### 2.1 수집한 데이터

**데이터셋 ① 국립중앙의료원 치매안심센터 정보 (256행)**
- 출처: 공공데이터포털 — https://www.data.go.kr/data/15138421/fileData.do
- 특징: 치매안심센터명, 시도, 시군구, 주소, 전화번호, 위경도 등 컬럼이 이미 정제되어 있는 데이터. 다만 전부 "치매안심센터" 유형만 존재하며, 운영기관·제공 프로그램 정보는 없음.

**데이터셋 ② 전국치매센터표준데이터 (317행)**
- 출처: 공공데이터포털 — https://www.data.go.kr/data/15021138/standard.do
- 특징: 행정안전부 표준데이터 형식을 따르는 데이터셋으로, 치매안심센터·광역치매센터·치매상담전화센터 3개 유형을 모두 포함한다. 다만 표준데이터 자체가 각 지자체가 제출한 다양한 소스를 취합한 것이라, 주소 표기·기관명 표기 등이 정제되지 않고 혼재되어 있다(예: 시도/시군구가 별도 컬럼이 아니라 도로명주소 문자열 안에 포함됨). 대신 이 데이터셋에만 운영기관 정보, 인력 현황, 제공 프로그램 목록이 담겨 있다.

### 2.2 프로젝트와의 관련성

챗봇이 "우리 동네 치매안심센터 어디야?", "이 센터에서 하는 프로그램 뭐 있어?" 같은 위치·기관 기반 질문에 답하려면 전국 센터의 위치, 운영기관, 제공 프로그램 정보가 구조화된 형태로 필요하다. 단일 데이터셋만으로는 정제된 주소(①)와 프로그램·인력 정보(②)를 동시에 얻을 수 없어 두 데이터셋을 병합했다.

### 2.3 수집 방법

공공데이터포털에서 CSV 파일을 직접 다운로드했다 (API 호출이 아닌 정적 파일 다운로드 방식).

### 2.4 전처리 과정

**2.4.1 컬럼 선별**

두 데이터셋에서 최종 스키마에 필요한 컬럼만 남기고, 중복되거나 결측이 심한 컬럼(도로명코드/법정동코드, 지번주소, 건축물면적, 운영위탁일자 등)은 제외했다.

**2.4.2 시도/시군구 정규화**

데이터셋 ②는 시도/시군구가 별도 컬럼이 아니라 도로명주소 문자열에 섞여 있어, 주소 앞 토큰을 파싱해 별도 컬럼으로 분리했다. 이 과정에서 "전라북도"/"전북특별자치도"처럼 두 데이터셋 간 표기가 다른 경우를 하나로 통일했고, 세종특별자치시처럼 시군구 단위가 없는 광역자치단체는 예외 처리했다.

**2.4.3 두 데이터셋 매칭·병합**

데이터셋 ①(256행, 정제된 이름)을 기준으로, 데이터셋 ②에서 대응하는 행을 찾아 운영기관·프로그램·인력 정보를 보강했다. 매칭은 1차로 센터명(접미어 제거 후 핵심 지역명 비교), 2차로 좌표 최근접 거리 기반으로 진행했다. 이 과정에서 데이터셋 ①은 전부 "치매안심센터" 유형이므로, 매칭 후보를 데이터셋 ②에서도 동일 유형("치매안심센터")으로 제한해 광역치매센터·치매상담전화센터와 잘못 매칭되는 것을 방지했다. 최종적으로 256개 중 214개가 매칭되었고, 매칭 안 된 나머지는 데이터셋별로 별도 노드로 유지했다.

**2.4.4 인력 현황 표준화**

데이터셋 ②의 `기타인원현황` 컬럼은 자유 텍스트(예: "임상심리사 1, 음악치료사 1")로 기재되어 있어, 15개 내외의 표준 직종 카테고리로 매핑 테이블을 만들어 `{직종: 인원수}` 형태로 파싱했다.

**2.4.5 매칭 오류 검증 및 수정**

초기 매칭 로직에서는 좌표 거리만으로 매칭할 경우, 근거리에 있는 광역치매센터가 일반 치매안심센터로 잘못 매칭되는 사례가 발견되었다(예: 경주시, 부산 서구, 세종시). 유형을 사전 필터링하는 방식으로 로직을 수정해 이 문제를 해결했다.

### 2.5 최종 결과

최종적으로 노드 5종(시도 17개, 시군구 269개, 치매안심센터 357개, 운영기관 290개, 프로그램 749개)과 관계 5종으로 구성된 Neo4j 그래프로 적재했다. 상세 스키마는 `DB설계.md` 참조.

## 3. VectorDB 데이터 — 증상/가이드라인 문서

### 3.1 수집한 데이터

치매 조기검진 절차 및 관련 법령 안내 문서를 대상으로 했다.

| 출처 | URL |
|---|---|
| 치매안심센터 (국립치매관리기관) | https://ansim.nid.or.kr/introduce/early_service.aspx |
| 보건복지부 | https://www.mohw.go.kr/menu.es?mid=a10712010100 |
| 법제처 찾기쉬운 생활법령정보 | https://www.easylaw.go.kr/CSP/OnhunqueansInfoRetrieve.laf?onhunqnaAstSeq=97&onhunqueSeq=4461 |

### 3.2 프로젝트와의 관련성

챗봇이 "치매 조기검진은 어떻게 받나요", "비용 지원이 있나요" 같은 절차·제도 관련 질문에 답하려면, 정형 데이터가 아닌 서술형 가이드라인 문서가 필요하다. 이 정보는 GraphDB로 표현하기 어려운 비정형 지식이라 별도로 VectorDB에 적재했다.

### 3.3 수집 방법

각 URL에 대해 `robots.txt`를 확인해 크롤링 허용 여부를 먼저 검사한 뒤, 허용된 페이지만 수집했다. 페이지별로 본문 영역의 CSS 셀렉터가 달라 사이트마다 셀렉터를 개별 지정해 본문 텍스트만 추출했다.

### 3.4 문서 정제 방법

수집된 원문에서 다음 기준으로 정제했다:
- 5자 미만의 짧은 줄 제거
- 네비게이션 문구("홈 > ...> ..."), 이전/다음/목록/인쇄/공유 버튼 텍스트, 저작권 문구를 정규식 패턴으로 제거
- 문서 내 중복 줄 제거
- 정제 후 전체 텍스트가 30자 이하인 문서는 정보량이 없다고 판단해 폐기

### 3.5 Chunking 방법 및 기준

`RecursiveCharacterTextSplitter`를 사용해 chunk_size=400자, overlap=50자로 분할했다. 구분자는 문단(`\n\n`) → 줄바꿈(`\n`) → 문장(`. `) → 어절(` `) 순으로 우선순위를 두어, 가능한 한 의미 단위(문단·문장)를 보존하며 분할되도록 했다. chunk_size/overlap 값은 세밀한 실험적 튜닝보다는 일반적으로 통용되는 범위(수백 자 단위, overlap은 chunk_size의 10~15% 수준)를 참고해 설정한 값이며, 추후 검색 품질 평가를 통해 조정 가능하다.

각 청크의 ID는 `출처 URL + 청크 순번 + 텍스트`를 해시한 결정론적 값으로 생성했다. 이 방식 덕분에 동일한 파이프라인을 재실행해도 같은 청크는 같은 ID를 가지므로, Qdrant에 upsert(덮어쓰기)로 적재되어 중복이 발생하지 않는다.

### 3.6 임베딩 처리

임베딩 모델은 `BAAI/bge-m3`(다국어 지원, dense 임베딩 1024차원)를 사용했으며, 코사인 유사도 기준으로 Qdrant 컬렉션(`dementia_guideline`)에 적재했다. 검색 시에는 유사도 0.4 미만인 결과는 관련도가 낮다고 판단해 제외한다.

## 4. 데이터 품질 검증

### 4.1 GraphDB 검증

- 노드/관계 CSV 개수와 AuraDB 적재 후 개수가 일치하는지 카운트 쿼리로 확인
- 매칭 로직 개선 전후로 유형(`유형` 프로퍼티) 오분류 사례를 표본 검증 (경주시, 부산 서구, 세종시)
- 지역·운영기관·프로그램 기준 조회 결과가 실제 원본 데이터와 일치하는지 교차 검증

### 4.2 VectorDB 검증

- 증상 관련 질의 샘플로 검색 결과의 관련도 점수 및 출처 확인

## 5. 한계 및 참고사항

- 데이터셋 ②(전국치매센터표준데이터)의 일부 행은 원본 자체에 오류가 있다 (예: 이름은 "OO치매안심센터"인데 유형 컬럼은 "광역치매센터"로 기재된 사례). 이는 원본 공공데이터의 한계로, 전처리 단계에서 발견했으나 임의로 수정하지 않고 원본 값을 그대로 유지했다.
- VectorDB의 chunk 크기·overlap·유사도 임계값은 초기 설정값으로, 실제 서비스 질의 패턴에 따라 추후 조정 여지가 있다.
