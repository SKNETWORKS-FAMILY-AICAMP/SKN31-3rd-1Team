# 데이터 수집 및 전처리 문서

## 1. 개요

### 1.1 데이터셋 소개

치매알람은 치매 초기 증상 안내 및 지역 치매안심센터 정보 제공을 위한 LangGraph 기반 RAG 챗봇이다. 보호자와 사용자가 자연어로 질문하면, **구조화된 지역/센터 정보**(GraphDB)와 **비정형 증상·가이드라인 문서**(VectorDB) **두 종류의 데이터를 사용·검색**하여 답변을 생성한다.

### 1.2 데이터 구성 개요

| 구분 | 데이터 | 형태 | 용도 |
|---|---|---|---|
| GraphDB | 전국 치매안심센터 정보 (2개 공공데이터셋 병합) | 구조화 (CSV → Neo4j) | 지역/운영기관/프로그램 기준 센터 조회 |
| VectorDB | 치매 조기검진·관련 법령 가이드라인 | 비정형 문서 (웹페이지 → 텍스트) | 증상/검진 절차 관련 의미 기반 검색 |

## 2. GraphDB 데이터 — 치매안심센터 정보

### 2.1 수집한 데이터

**데이터셋 ① 국립중앙의료원 치매안심센터 정보 (256행)**
- 출처: 공공데이터포털 — https://www.data.go.kr/data/15138421/fileData.do
- 특징: <b>치매안심센터명</b>, 시도, 시군구, 주소, 전화번호, 위경도 등 컬럼이 이미 정제되어 있는 데이터. 다만 전부 "치매안심센터" 유형만 존재하며, 운영기관·제공 프로그램 정보는 없음.

**데이터셋 ② 전국치매센터표준데이터 (317행)**
- 출처: 공공데이터포털 — https://www.data.go.kr/data/15021138/standard.do
- 특징: 행정안전부 표준데이터 형식을 따르는 데이터셋으로, <b>치매안심센터·광역치매센터·치매상담전화센터 3개 유형</b>을 모두 포함한다. 다만 표준데이터 자체가 각 지자체가 제출한 다양한 소스를 취합한 것이라, 주소 표기·기관명 표기 등이 정제되지 않고 혼재되어 있다(예: 시도/시군구가 별도 컬럼이 아니라 도로명주소 문자열 안에 포함됨). 
- 대신 이 데이터셋에만 <b>운영기관 정보, 인력 현황, 제공 프로그램 목록</b>이 담겨 있다. 따라서 첫번째 데이터셋을 메인으로 하되, 필요한 정보를 표준데이터를 기반으로 추가한다.
- 단일 데이터셋만으로는 정제된 주소(①)와 프로그램·인력 정보(②) 등을 동시에 얻을 수 없어 두 데이터셋을 병합했다.

### 2.2 프로젝트와의 관련성 - GraphDB 선정 이유📌

GraphDB를 사용한 이유는 노드와 관계를 탐색하는 질의에 강하기 때문이다.

예를 들어, 챗봇이 "우리 동네 치매안심센터 어디야?", "이 센터에서 하는 프로그램 뭐 있어?" 같은 위치·기관 기반 질문에 답하려면 전국 센터의 위치, 운영기관, 제공 프로그램 정보가 구조화된 형태로 필요하다. 

결론적으로 GraphDB 사용하여, "서울시 치매안심센터는 몇 개인가?", "의사가 가장 많은 센터는?", "어떤 기관이 운영하는가?"처럼 관계 기반 질의를 빠르게 처리할 수 있다.

### 2.4 전처리 과정

**2.4.1 컬럼 선별**

두 데이터셋에서 최종 스키마에 필요한 컬럼만 남기고, 중복되거나 결측이 심한 컬럼(도로명코드/법정동코드, 지번주소, 건축물면적, 운영위탁일자 등)은 제외했다.

**2.4.2 시도/시군구 정규화**

데이터셋 ②는 시도/시군구가 별도 컬럼이 아니라 도로명주소 문자열에 섞여 있어, 주소 앞 토큰을 파싱해 별도 컬럼으로 분리했다. 이 과정에서 "전라북도"/"전북특별자치도"처럼 두 데이터셋 간 표기가 다른 경우를 하나로 통일했고, 세종특별자치시처럼 시군구 단위가 없는 광역자치단체는 예외 처리했다.

**2.4.3 두 데이터셋 매칭·병합**

데이터셋 ①(256행, 정제된 이름)을 기준으로, 데이터셋 ②에서 대응하는 행을 찾아 운영기관·프로그램·인력 정보를 보강했다. 매칭은 1차로 센터명(접미어 제거 후 핵심 지역명 비교), 2차로 좌표 최근접 거리 기반으로 진행했다. 이 과정에서 데이터셋 ①은 전부 "치매안심센터" 유형이므로, 매칭 후보를 데이터셋 ②에서도 동일 유형("치매안심센터")으로 제한해 광역치매센터·치매상담전화센터와 잘못 매칭되는 것을 방지했다. 최종적으로 256개 중 214개가 매칭되었고, 매칭 안 된 나머지는 데이터셋별로 별도 노드로 유지했다.

**2.4.4 인력 현황 표준화**

데이터셋 ②의 `기타인원현황` 컬럼은 자유 텍스트(예: "임상심리사 1, 음악치료사 1")로 기재되어 있어, 15개 내외의 표준 직종 카테고리로 매핑 테이블을 만들어 `{직종: 인원수}` 형태로 파싱했다.

**2.4.5 매칭 오류 검증 및 수정**

- **중복 노드 제거**
  - 원본 CSV 2종 병합 시 이름 표기 차이로 중복 발생 ex. 광주서구안심센터 or 서구안심센터  
  - 전화번호를 기준으로 **45개 그룹(약 90개 노드)** 식별 및 병합

- **홈페이지 데이터 보완**
  - **치매안심센터 홈페이지** 결측 **101건** 확인
  - 전수 조사를 통해 **`nid.or.kr` 패턴** 및 **지자체 공식 도메인**으로 수동 매핑
  - 데이터 품질 향상을 위해 **운영기관 홈페이지 225건**을 추가 조사하여 수동 적재


- **데이터 타입 정규화**
  - 의사·간호사·사회복지사 인원수 및 우편번호를 **float → int** 변환

- **URL 정제**
  - 누락된 **`https://` 프로토콜**을 일괄 보완

### 3.2 프로젝트와의 관련성 -VectorDB 선정 이유 📌

- **증상이나 치매 정보**를 **의미 기반**으로 검색하기 위해 사용하였다. 사용자가 표현을 달리해도 의미가 유사한 문서를 찾아 답변할 수 있다.

- 예를 들어, 사용자가 "최근 자꾸 같은 말을 반복해요", "치매 검사는 어떻게 받나요?" 와 같이 증상이나 치매 관련 질문을 입력하면, AI가 관련성이 높은 전문 자료를 검색(RAG) 하여 근거 기반의 답변을 제공할 수 있도록 VectorDB를 구축하였다.

- 수집 대상은 치매안심센터, 보건복지부, 질병관리청, 법제처, 대학병원 등 공신력 있는 기관의 안내 문서이며, 이를 크롤링한 후 청킹(Chunking) 및 임베딩(Embedding) 하여 Qdrant에 저장하였다.

- **논문이나 학술자료는 전문 용어가 많아 일반 사용자가 이해하기 어려운 경우**가 많다. 반면 공공기관과 의료기관의 홈페이지는 **일반 시민을 대상으로 작성된 안내 자료**이므로, 사용자에게 이해하기 쉬우면서도 객관적인 정보를 제공하기에 적합하여 RAG 데이터로 활용하였다.

### 3.3 수집 방법

각 URL에 대해 `robots.txt`를 확인하여 크롤링이 허용된 사이트만 대상으로 수집하였다. 웹사이트마다 본문이 위치한 HTML 구조가 달라, 사이트별로 본문 영역의 CSS Selector를 개별 설정하여 메뉴나 광고를 제외한 실제 안내 내용만 추출하였다.

### 3.4 문서 정제 방법

수집한 문서의 품질을 높이기 위해 다음과 같은 전처리를 수행하였다.

- 5자 미만의 불필요한 문장 제거
- 메뉴, 이전/다음 버튼, 공유 버튼, 저작권 문구 등 본문과 관계없는 텍스트 제거
- 문서 내 중복된 문장 제거
- 정제 후 내용이 30자 이하인 문서는 정보가 부족하다고 판단하여 제외

### 3.5 Chunking 방법 및 기준

긴 문서를 검색하기 쉽도록 **`RecursiveCharacterTextSplitter`**를 사용하여 문서를 여러 개의 작은 단위(Chunk)로 분할하였다.

- Chunk 크기: 400자
- Overlap: 50자

문서를 나눌 때는 의미가 끊기지 않도록 다음 순서로 분할 기준을 적용하였다.

① 문단 → ② 줄바꿈 → ③ 문장 → ④ 단어

또한 동일한 문서를 여러 번 적재하더라도 중복이 발생하지 않도록 URL, 청크 순번, 텍스트를 기반으로 고유 ID를 생성하였다. 이 방식으로 Qdrant에 다시 저장해도 기존 데이터가 갱신되며 중복 데이터는 생성되지 않는다.

각 청크에는 **다음 메타데이터**를 함께 저장하였다.

| 필드 | 설명 |
|---|---|
| `text` | 청크 본문 |
| `title` | 출처 페이지 제목 |
| `source_url` | 출처 URL |
| `source_type` | 문서 유형 (`guideline` 고정) |
| `chunk_index` | 문서 내 청크 순번 |
| `total_chunks` | 해당 문서의 전체 청크 수 |

### 3.6 임베딩 및 VectorDB 구축 📌

문서를 벡터로 변환하기 위해 OpenAI의 **text-embedding-3-small 모델**을 사용하였다.

초기에는 **BAAI/bge-m3**를 사용했으나, **OpenAI API 기반 서비스와의 연동 편의성 및 검색 성능**을 고려하여 **text-embedding-3-small**으로 변경하였다. 

생성된 임베딩은 Qdrant(`dementia_guideline`) 컬렉션에 저장하였으며, 검색 시에는 코사인 유사도(Cosine Similarity)를 이용하여 사용자 질문과 가장 의미가 가까운 문서를 찾는다.

효율적인 처리를 위해 100개의 청크씩 배치(Batch) 처리하여 API 호출 횟수를 줄였으며, 검색 결과 중 유사도 0.3 미만인 문서는 관련성이 낮다고 판단하여 제외하였다.

**VectorDB Tool 구성** 📌

Qdrant에 저장된 임베딩 데이터를 검색하기 위해 Vector Search Tool을 구현하였다.

**LangChain Retriever** 대신 **Qdrant Client와 OpenAI 라이브러리**를 직접 사용하여 검색 로직을 구현하였으며, 
이를 통해 유사도 임계값(Threshold), Top-K 개수, 출력 형식을 서비스 목적에 맞게 세밀하게 제어할 수 있도록 하였다.


- 상세 설계 및 TOOL은 **[시스템아키텍쳐.md](산출물/시스템아키텍쳐.md)** 를 참고한다.

## 4. 데이터 품질 검증

### 4.1 GraphDB 검증

- CSV 데이터의 **노드와 관계 개수**가 Neo4j AuraDB에 정상적으로 저장되었는지 확인
- 매칭 로직 개선 전후를 비교하여 **유형 분류가 올바르게 이루어졌는지** 표본 검증 (경주시, 부산 서구, 세종시)
- 지역, 운영기관, 프로그램 조회 결과가 **원본 데이터와 일치하는지** 교차 검증
- 9개의 고정 Tool과 `flexible_graph_search`의 검색 결과가 **정답과 얼마나 일치하는지** Precision과 Recall을 통해 평가

### 4.2 VectorDB 검증

- 증상, 법령, 서비스 관련 질의를 통해 **검색된 문서의 관련도와 출처** 확인
- 질의-정답 문서 쌍을 이용하여 **Top-k 검색 성능(Hit Rate, MRR)** 평가

### 4.3 RAG 성능 평가 (RAGAS)

- **RAGAS**를 활용하여 **Faithfulness**, **Answer Relevancy**, **Context Precision**, **Context Recall** 평가
- 대표 질문을 선정하여 **정확성, 유용성, 표현의 자연스러움, 일관성**을 기준으로 정성 평가 수행

- 상세 결과는 **[성능평가.md](산출물/성능평가.md)** 를 참고한다.
