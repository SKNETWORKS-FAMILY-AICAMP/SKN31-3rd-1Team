## 1. GraphDB 성능 평가

### GraphDB(Neo4j)를 사용한 이유

센터-프로그램-지역-운영기관 간 **다대다 관계**를 다루는 질문이 많아,
JOIN 기반 RDB보다 관계 자체를 1급 시민으로 다루는 GraphDB가 적합하다고 판단.

**예시 질문**
- "강남구에 치매안심센터 몇 개야?" → 지역(시군구)-센터 관계 집계
- "인지훈련 프로그램 하는 센터 어디어디야?" → 센터-프로그램 다대다 관계 탐색
- "삼성서울병원이 운영하는 센터는?" → 운영기관-센터 관계 탐색
- "의사가 2명 이상인 센터는?" → 속성 조건 + 관계 결합 질의

이런 질문들은 RDB로 하면 JOIN 테이블을 여러 개 거쳐야 하지만, GraphDB는
관계를 그대로 순회(traversal)하면 되어 쿼리가 직관적이고 빠름.

**실험 노트북**: `graph_db/eval_graphdb.ipynb`

### 평가 방법

질의 10개 세트(9개 고정 tool 각 1건 + flexible_graph_search fallback 1건)에 대해
tool 실행 결과 텍스트에 Ground Truth 키워드가 모두 포함되는지로 Precision/Recall 산출.

### 결과

|---|---|---|---|---|

### 코멘트

10개 질의 모두 고정 tool 라우팅 및 fallback 모두 정답 키워드를 정확히 포함해 반환.
9개 고정 tool의 라우팅 로직과 flexible_graph_search fallback 모두 의도대로 동작함을 확인.

다만 평가 방식이 "결과 텍스트에 키워드가 포함되는가"를 기준으로 하는 다소 관대한 기준이라,
결과의 완전한 정확성보다는 정답 항목이 결과에서 누락되지 않았는지를 보는 지표에 가까움 —
발표 Q&A에서 평가 방법론 관련 질문이 나올 수 있어 이 점은 짚고 넘어가는 게 좋을 듯.

## 2. VectorDB 성능 평가 (유사도)

### VectorDB(Qdrant)를 사용한 이유

치매 증상/예방 관련 비정형 가이드라인 문서(치매 조기 증상, 대응 방법 등)는
정형화된 관계가 아니라 **의미적 유사도 기반 검색**이 필요해, 임베딩 기반
VectorDB가 적합하다고 판단.

**예시 질문**
- "요즘 자꾸 물건을 어디 뒀는지 까먹어요, 이거 치매 초기 증상인가요?" → 구조화 안 된 자연어 증상 설명을 문서와 의미 매칭
- "부모님이 치매 진단받으면 뭐부터 해야 하나요?" → 특정 키워드 매칭이 아니라 맥락 이해 필요
- "경도인지장애랑 치매랑 뭐가 달라요?" → 개념 설명형 질문, 정확한 텍스트 일치가 아닌 유사 문맥 검색

이런 질문은 GraphDB의 정형 관계 탐색으로는 답할 수 없고, 문서 내용 자체의
의미 유사도로 찾아야 하기 때문에 VectorDB를 병행.

### 임베딩 모델 선택 이유

- 초기에는 오픈소스 임베딩 모델(`BAAI/bge-m3`)을 검토했으나, 이를 서빙할
  **무료 서버가 없었고**, 직접 GPU 서버를 띄우기엔 비용 부담이 커서 채택하지 않음
- 최종적으로 **OpenAI 임베딩 모델**로 변경. 별도 서버·GPU 없이 API 호출만으로
  바로 사용 가능해 부트캠프/해커톤 일정상 인프라 구축 부담을 줄일 수 있었음
- (참고: `EMBEDDING_DIMS` 등 config 설정값은 실제 사용 모델에 맞게 재확인 필요)

**실험 노트북**: `vector_db/eval_vectordb.ipynb`

### 평가 방법

질의-정답 키워드 쌍 10개 세트에 대해 top-k 검색 결과 청크에 키워드가 등장하는
첫 순위(rank)를 찾아 Hit Rate / MRR 산출. `SCORE_THRESHOLD=0.3`, `TOP_K=4` 기준.

### 결과

| 지표 | 값 |
|---|---|
| Hit Rate | 1.0000 (10/10) |
| MRR | 0.9333 |

| 질문 | Hit | Rank | MRR |
|---|---|---|---|
| 부모님이 자꾸 같은 말을 반복해요, 치매인가요? | ✅ | 1 | 1.0000 |

| 치매 환자 가족으로서 어떻게 대해야 하나요? | ✅ | 1 | 1.0000 |

### 코멘트

10문항 모두 hit는 했으나("치매 예방" 질의만 3위로 밀림), 나머지 9개는 전부 1위로 정확히 매칭됨.
"치매 예방" 질의의 정답 청크가 top-1이 아니었던 이유는, 실제 검색된 문서 제목을 확인해보니
다른 9개 문항은 전부 "치매환자의 가족을 위한 정보 | 국가건강정보포털"류 문서에서 매칭된 반면
이 문항만 "치매(Dementia) | 질환백과 | 서울아산병원" 문서에서 매칭됨 — 출처 자체가 달랐던 게
순위가 밀린 원인으로 보임.

추가로, 매칭된 키워드 개수를 함께 확인해보니 10문항 중 1건("부모님이 자꾸 같은 말을...")은
정답 키워드 3개 중 컬렉션 전체에 흔한 "치매"만 걸려서 통과한 경우였음(약한 매칭). 이 문항을
제외한 나머지 9건은 구체적인 키워드(검진, 알츠하이머, 가족, 비용 등)로 매칭돼 신뢰도가 높음.
전반적으로 TOP_K=4, threshold=0.3 조합은 지금 질의셋 기준으로는 잘 작동하는 편이나, 평가
자체가 "정답 문서 ID 지정"이 아닌 "키워드 텍스트 매칭" 방식이라는 한계는 있음.

### 3-1. 정량 평가 (RAGAS)

#### 평가 방법

RAGAS `TestsetGenerator`로 Qdrant 청크 기반 테스트셋 자동 생성 후,
`get_structured_answer`로 응답을 생성해 Faithfulness, Answer Relevancy,
Context Precision, Context Recall 4개 지표 측정 (testset_size=15 요청, 실행 결과 15개 확보).

#### 결과

지난번 커널 재시작 이슈(vertexai mock 누락) 해결 후 재실행 성공. testset 15개 기준 결과:

| 지표 | 점수 |
|---|---|
| Faithfulness | 0.2657 |
| Answer Relevancy | 0.4088 |
| Context Precision | 0.4500 |
| Context Recall | 0.6000 |

![RAGAS 평가 결과](./images/ragas_result.png)

#### 코멘트

**4개 지표 다 낮은 편, 특히 Faithfulness(0.2657)가 가장 낮음.** 아래는 원인 분석이며,
아직 확정이 아니라 **추정 + 반박 증거가 섞인 상태**임을 감안하고 볼 것.

- **Faithfulness = 0.0인 문항이 15개 중 9개**, 그중 **7개는 Answer Relevancy도 NaN**으로 함께 잡힘.
  `eval.ipynb` 응답 생성 로직상 `structured["type"] == "choices"`(정보 부족 시 되묻기)인 경우
  되묻는 질문 문장을 그대로 "답변"으로 RAGAS에 넘기게 되어 있어, 검증할 주장 자체가 없어
  Faithfulness/Answer Relevancy가 구조적으로 0/NaN이 될 수 있음 — 이 코드는 이미 수정해서
  `response_type`을 남기고 reply만 따로 채점하도록 `eval.ipynb`에 반영해둠(재실행 필요).
- **다만 이 가설이 전부를 설명하진 않음.** "치매 환자에게 작업 요법이 필요한 이유는?" 문항은
  Faithfulness=0/Answer Relevancy=NaN으로 나왔지만, 온라인 정성평가 자료를 보면 되묻기가 아니라
  참고자료 2건을 인용한 정상적인 답변이 나갔음(아래 정성평가 예시 참고). 즉 choices 오염 외에도
  RAGAS의 자동 생성 reference(정답지)와 실제 검색·생성 내용이 어긋나는 별도 원인이 섞여 있을
  가능성이 있음 — 재실행 결과를 봐야 choices 비중이 실제로 얼마나 되는지 확정할 수 있음.
- Context Precision(0.45)·Context Recall(0.60)은 낮긴 해도 0~1 사이에 고르게 분포하고,
  GraphDB/VectorDB 검색 성능은 위에서 별도로 검증됐으므로, 검색 자체가 망가진 신호는 아님.
- 정성평가(아래 3-2) 예시 2건 모두 실제로는 톤·근거 인용이 준수한 답변이었음 — RAGAS 점수와
  실제 체감 품질 사이에 괴리가 있다는 정황.

**결론(발표용 한 줄)**: "RAGAS 지표는 낮게 나왔지만, 채점 파이프라인이 되묻기 응답을 답변으로
잘못 채점한 버그를 발견해 수정했고(재실행 대기), 정성평가로 본 실제 답변 품질은 이 점수보다
높다" — 정도가 지금까지 확인된 사실 기준으로 방어 가능한 선. "검색은 100%니까 생성도 사실
괜찮다"처럼 검증 안 된 논리로 비약하지 않는 게 안전함.

### 3-2. 정성 평가

#### 평가 방법

기존에 진행한 **온라인 정성평가** 자료 중 2건을 반영. 실제 배포 환경에서 질문을 입력해 받은
응답을 팀 관점에서 채점. RAGAS 같은 자동 지표가 못 잡는 "실제로 유용한가", "톤이 적절한가"를
확인하는 목적이며, 공교롭게도 이 2건은 RAGAS 정량 평가에서도 낮은 점수가 나온 문항이라
정량-정성 비교 사례로도 같이 활용.

#### 평가 기준 (예시)

| 항목 | 설명 |
|---|---|
| 정확성 | 사실과 다른 내용이 없는가 |
| 유용성 | 실제 사용자(보호자/환자)에게 실질적으로 도움이 되는가 |
| 톤/표현 | 민감한 주제(치매)를 다루는 데 있어 표현이 적절한가 |
| 일관성 | 비슷한 질문에 답변이 오락가락하지 않는가 |

#### 결과

![정성평가 예시 1 - 작업요법](./images/qualitative_example_1_작업요법.png)

![정성평가 예시 2 - 베란다 소변](./images/qualitative_example_2_베란다소변.png)

| 질문 | RAGAS 점수 | 정확성 | 유용성 | 톤/표현 | 일관성 | 종합 코멘트 |
|---|---|---|---|---|---|---|
| 치매 환자에게 작업 요법이 필요한 이유는 무엇인가요? | Faithfulness 0.0, Answer Relevancy NaN | 5 | 4 | 5 | 판단보류* | 참고자료 2건(서울아산병원, 분당서울대병원) 인용하며 목적·효과를 정확히 설명. 어르신 관심사를 되묻고 지역 센터 연계까지 자연스럽게 제안. RAGAS 점수와 실제 품질이 크게 어긋난 사례 |
| 치매 환자가 베란다에서 소변 보는 이유가 뭐에요? | Faithfulness 0.5, Context Recall 0.0 | 4 | 4 | 5 | 판단보류* | 화장실 혼동 등 원인을 무리 없이 설명, 표식 부착 등 실행 가능한 대처법 제시. "환자분의 안전과 보호자님의 고충이 크실 것 같다"는 공감 표현 적절. 참고자료 인용 있음 |

*일관성은 문항당 1회 대화만 확인해 판단 불가 — 같은 질문을 톤/표현 바꿔 2~3회 반복 테스트해야
제대로 채점 가능. 시간 남으면 추가할 항목.

#### 코멘트

두 예시 모두 **정량 지표(RAGAS)는 낮았지만 정성적으로는 준수한 답변**이었음 — 특히 "작업 요법"
문항은 Faithfulness가 0.0으로 나왔는데도 실제로는 출처를 인용한 정상 답변이었어서, RAGAS
자동 채점과 실제 체감 품질이 꽤 벌어질 수 있다는 걸 보여주는 사례. 다만 예시가 2건뿐이라
"RAGAS가 늘 이런 식으로 저평가한다"고 일반화하긴 이름 — 표본을 늘리거나(5~6문항 이상),
`eval.ipynb` 수정본을 재실행해 `response_type`별로 나눠서 보는 게 더 확실함.

## 4. 종합 요약

**검색(retrieval) 단은 GraphDB·VectorDB 모두 확인 완료, 결과 우수.** GraphDB는 9개 고정 tool +
flexible fallback 10개 케이스 전부 Precision/Recall 1.0으로, 라우팅 로직이 의도대로
작동함을 확인. VectorDB는 Hit Rate 1.0(10/10), MRR 0.9333으로 대부분 top-1 매칭이며
"치매 예방" 질의 1건만 3위로 밀림(출처 문서 자체가 달랐던 게 원인으로 보임) — 발표에서
"검색은 검증됐다"는 포인트로 쓸 수 있음.

**생성(generation) 단, RAGAS는 완료됐으나 지표 자체는 낮음(Faithfulness 0.2657 등).** 15개
문항 중 9개가 Faithfulness=0, 그중 7개가 Answer Relevancy=NaN으로 함께 잡히는 패턴을 보면,
"정보 부족 시 되묻는(`choices`) 응답이 답변 자리에 그대로 채점된" 파이프라인 문제가 일부
원인으로 보이고, 이 부분은 `eval.ipynb`를 이미 수정해뒀음(재실행 대기). 다만 온라인
정성평가 자료로 대조해보니 이 가설만으론 다 설명이 안 됨 — Faithfulness=0으로 나온 문항
중 하나("작업 요법" 질문)는 실제로는 되묻기가 아니라 출처를 인용한 정상 답변이었음. 즉
RAGAS 저평가에는 채점
버그 외에 다른 원인(자동 생성된 reference와 실제 응답의 불일치 등)도 섞여 있을 가능성이
있고, 정확한 그림은 수정된 코드로 재실행해봐야 나옴.

**발표 권장 멘트**: "검색(GraphDB·VectorDB)은 정량적으로 검증됨. RAGAS 생성 지표는 낮게
나왔는데, 그 중 채점 파이프라인이 되묻기 응답을 답변으로 잘못 채점하던 버그를 발견해 수정했고
(재실행 대기 중), 실제 정성평가로 확인한 답변 품질은 그 점수보다 높았다." — 검증된 사실
기준으로 이 정도가 방어 가능한 선. "검색이 1.0이니 생성도 사실 괜찮을 것"이라는 식의 비약은
피하는 게 안전함.

**참고**: `overall_summary.png`(종합평가 인포그래픽)를 슬라이드에 그대로 쓸 거면 "RAGAS
평가에서는 Context Recall이 가장 높은 점수를 보여 전반적으로 안정적인 성능을 확인했다"는
문구는 다시 보는 게 좋음 — 4개 지표 중 3개가 0.5 미만인 상태에서 "안정적"이라는 표현은
Q&A에서 반박당하기 쉬움. 숫자(GraphDB/VectorDB 카드)는 그대로 쓰되, RAGAS 파트 문구만
위 "발표 권장 멘트"로 바꿔 쓰는 걸 추천.
